## Active / next — SP3a (solver core + beam search)
**Read the SP3a spec (source of truth) first; then run `writing-plans`.** What SP3a builds:
- New **pure, DI'd** `src/lib/solver/` module. Imports `@/lib/validation` (`validateBuild`, `evaluateArtifactCapacity`, `canAddArtifactPerk`), `@/lib/synergy` (`scoreSynergy`, `getSynergies`), `@/lib/types`. Reaches synergy ONLY through the SP1 seam. No filesystem (inject `Lookup` + `Indexes`). Deterministic. Layering one-way (nothing in validation/synergy imports solver).
- `solve(build, ctx, options?) → SolveResult` where `ctx = { lookup, indexes }`, `options = { beamWidth?=16, topN?=5 }`, `SolveResult = { builds: RankedBuild[]; feasible: boolean }`, `RankedBuild = { build; score; synergy: SynergyScore; statFit }`.
- **Input contract:** user pins `subclass.element`+`aspectHashes` (fixes fragment-slot cap) and `artifact.artifactHash` (fixes perk pool); solver fills the fragment set + artifact perk set (two OPEN dimensions). Pre-present fragments/perks are kept.
- **Beam mechanism:** state = partial `Build`; move = add one fragment OR one artifact perk (fragments gated by slot cap; perks gated by `canAddArtifactPerk`). Prune by `realized(scoreSynergy) + optimisticBound`; final ranking of complete builds by realized `scoreSynergy.score + statFit`.
- **Optimistic bound** = Σ over each unmatched produced/consumed keyword of its best-possible future chain link, credited ONLY if an OPEN dimension's pool can supply the complementary side (keyword index). Must be **admissible** (never underestimate) — that is the correctness property; tightness is a later tuning follow-up.
- **`StatFit` seam:** `interface StatFit { (build, ctx): number }` + stub `neutralStatFit = () => 0`. SP4 fills it. v1 ranking = synergy-only.

**The test that proves SP3a done (name it in the plan, don't just "add tests"):** the **delayed-reward** test — a producer fragment whose ONLY consumer is an artifact perk must survive the beam and appear in the top-ranked build; the test must FAIL if the optimistic bound is removed (a realized-only beam prunes the producer). Plus: bound-is-an-upper-bound on synthetic pools; candidate-gen respects cap + oracle; determinism across permuted input; real-dataset integration (coupled pair outranks uncoupled, non-empty `why`).

## Decisions resolved (do not relitigate)
- **SP3 is sliced; SP3a = fragments + artifact perks.** *Why:* the dominant risk is beam-search-vs-pairwise-synergy (delayed reward). A slice of two *synergy-coupled* subset-dimensions forces the beam to solve delayed reward on a small surface, before scaling all dimensions in SP3b. Not one independent-dimension pair (proves only plumbing), not one dimension (cross-dim synergy never appears).
- **SP3a partial scoring = optimistic completion bound (admissible upper bound).** *Why:* principled fix for delayed reward — never prunes a producer whose consumer could still come; it is the mechanism SP3b scales. Chosen over a crude tunable potential term and over reduce-then-join (dodges partial scoring, K^d blows up later). Correctness needs only admissibility; bound *tightness* is a later tuning follow-up (beam width absorbs looseness).
- **SP3a pins subclass (element+aspects) + artifact; solver fills fragment set + perk set.** *Why:* fixes the fragment-slot cap + perk pool → exactly two open subset-dimensions. Solver-chosen artifact (7) and solver-chosen aspects (dynamic cap) are SP3b.
- **SP3a ranking = synergy + stubbed stat-fit (`neutralStatFit`→0).** *Why:* stat-fit is SP4 (armor optimizer). Define the `StatFit` seam now, fill later, without reworking the solver.
- **Solver reaches synergy ONLY via `scoreSynergy`/`getSynergies`** (design's critical seam). *Why:* keeps the Phase-3 ML/embedding layer drop-in.
- **Phase 1 validator = rule registry + DI.** Pure `(build, lookup) => Violation[]` rules grouped per domain, concatenated by `validateBuild`; narrow `Lookup` seam so unit tests inject stubs. *Why:* isolated/testable, mirrors ingestion's small-module style.
- **Violation `category`: `game` vs `policy`.** All v1 rules are `game` (hard → `valid=false`). `policy` reserved for Phase 2 soft/synergy preferences (rank, don't invalidate). *Why:* clean seam without a fake split now.
- **Partial-build semantics:** an incomplete build is never "invalid"; each rule has a firing condition (fires only once its section is engaged). Empty/unselected rows never fire (guarded in weapon slot-uniqueness + perk constraints, armor slot-uniqueness, artifact pinning). *Why:* a build canvas mid-edit shouldn't be spammed.
- **User's baseline constraints as `game` floors** pairing with game ceilings → aspects **=2**, fragments **=max**, exotic armor **=1**, artifact tiers filled to `slots`. Ammo: **no double-primary; double-special allowed** (≥1 Special). *Why:* common-sense requirements that also prune the Phase 2 solver.
- **Dummy items excluded from classification.** Manifest has dummy weapon/armor copies (itemCategory "Dummies", `itemType` 20) with bogus data. `classify.ts` `isDummy` rejects them → weapons 2481→2208, armor 7551→6029, and "non-Power weapons never Heavy" holds. *Why:* real data-quality bug.
- **Artifacts — CEILING/cumulative model (CONFIRMED by user 2026-07-23).** 7 artifacts from `DestinyInventoryItemDefinition` "Artifact" items (bucket `1506418338`, 8 sockets), NOT `DestinyArtifactDefinition`. **3 tiers, socket ceilings 2/3/2 = 7 equipped, no duplicates.** Tiers are a **ceiling**: a higher-tier socket accepts its own perks **plus every lower tier's perks**, so the derived `perks` pools are **cumulative** (7/14/21) — the same perk hash appears in every tier at/above where it unlocks. Capacity legality from a flat list is a nested feasibility/matching problem, NOT a per-tier count. Memory: `artifact-tier-pools-cumulative`.
- **Artifacts — SP2 model decision (CONFIRMED by user 2026-07-23).** `Build.artifact` **stays flat** (`selectedPerkHashes`); NO per-tier/socket rework. *Why:* sockets within a tier are fungible and both consumers (synergy engine, solver) read only *which* perks are active, not placement — driver is solver **feasibility only**. The Phase-1 nested-ceiling walk is **already exact Hall feasibility** (its lowest-tier step subsumes the total-count check), so "sound-but-partial" was an understatement — it's complete for feasibility; only an explicit socket *assignment* is absent, which nothing needs. SP2 therefore extracted the logic into a **pure precompute+fast-eval oracle** (`buildCapacityModel`/`evaluate`/`canAdd`) shared by validator + solver, not a data-model change.
